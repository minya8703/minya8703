📐 SOLID 원칙

Q1. SOLID 원칙에 대해 설명하고, 실제 프로젝트에서 어떻게 적용했나요?

SOLID 원칙 개요:

  • S (Single Responsibility): 하나의 클래스는 하나의 책임만
  • O (Open-Closed): 확장에는 열려있고 수정에는 닫혀있어야 함
  • L (Liskov Substitution): 자식 클래스는 부모 클래스를 대체할 수 있어야 함
  • I (Interface Segregation): 클라이언트는 사용하지 않는 인터페이스에 의존하지 않아야 함
  • D (Dependency Inversion): 추상화에 의존해야지, 구체화에 의존하면 안됨

📌 실제 적용 사례 - MSA 세금계산서 Solution:

  • SRP 적용: JCo Server(SAP 연동), 중계 Server(데이터 변환), 회계 Server(홈택스 연동)로 책임 분리
    • 각 서비스가 하나의 명확한 책임만 가지도록 설계
    • 변경 사항이 다른 서비스에 영향을 주지 않음
  • OCP 적용: REST API 인터페이스로 서비스 간 통신
    • 새로운 회계 시스템 추가 시 중계 서버 수정 없이 확장 가능
    • 인터페이스는 고정, 구현체만 추가
  • DIP 적용: Spring의 의존성 주입 활용
    • 구체적인 구현체가 아닌 인터페이스에 의존
    • 테스트 시 Mock 객체로 쉽게 대체 가능

결과: 독립 배포 가능, 배포 시간 30분→10분, 유지보수성 향상

🗄️ DB 최적화

Q5. DB 성능 최적화를 위해 어떤 방법들을 사용했나요?

DB 최적화 기법:

1. 인덱스 설계 및 최적화
  • 적절한 인덱스 생성:
    • WHERE 절에 자주 사용되는 컬럼에 인덱스
    • JOIN 조건 컬럼에 인덱스
    • 카디널리티가 높은 컬럼 우선
  • 📌 한샘 SAP 연동:
    • 주문 조회 쿼리: 고객ID + 주문일자 복합 인덱스 생성
    • 조회 시간: 5초 → 0.3초 (약 94% 개선)
2. 쿼리 최적화
  • N+1 문제 해결:
    • Lazy Loading → Eager Loading (필요 시)
    • JPA Fetch Join 활용
  • SELECT 최적화:
    • SELECT * 대신 필요한 컬럼만 조회
    • 불필요한 데이터 전송 감소
  • 📌 MSA 세금계산서 프로젝트:
    • 세금계산서 조회: JOIN 5개 → 2개로 최적화
    • 필요한 데이터만 DTO로 projection
    • 처리 시간: 5분 → 1.5분 (70% 단축)
3. 배치 처리 최적화
  • Bulk Insert/Update:
    • 개별 INSERT 대신 배치 INSERT
    • JDBC Batch Size 설정 (보통 100~1000)
  • 📌 규슈 전력 청구서 생성:
    • 10만 건 청구서 생성 시 Bulk Insert 활용
    • 처리 시간: 30분 → 5분
4. 커넥션 풀 관리
  • HikariCP 설정:
    • maximumPoolSize: CPU 코어 수 * 2 + 디스크 수
    • minimumIdle: 기본 트래픽에 맞게 설정
    • connectionTimeout: 적절한 타임아웃 설정
5. 실행 계획 분석
  • EXPLAIN 활용:
    • 쿼리 실행 계획 분석
    • Full Table Scan 발견 시 인덱스 추가
    • 비효율적인 JOIN 방식 개선

Q6. 인덱스 설계 시 고려사항은 무엇인가요? 과도한 인덱스의 단점은?

인덱스 설계 고려사항:

  • 카디널리티: 고유값이 많을수록 인덱스 효과 ↑
    • 좋음: 주민등록번호, 이메일, 주문번호
    • 나쁨: 성별(M/F), Boolean 값
  • 선택도(Selectivity): 인덱스로 필터링되는 데이터 비율
    • 선택도 낮을수록 (결과 적을수록) 효과적
  • 복합 인덱스 순서: 카디널리티 높은 컬럼 먼저
    • INDEX(고객ID, 주문일자) > INDEX(주문일자, 고객ID)
  • WHERE/JOIN 분석: 자주 사용되는 조건절 우선

과도한 인덱스의 단점:

  • INSERT/UPDATE/DELETE 성능 저하:
    • 데이터 수정 시 모든 인덱스도 업데이트 필요
    • 인덱스 5개 → INSERT 시간 5배 증가 가능
  • 저장 공간 증가: 인덱스도 디스크 공간 차지
  • 옵티마이저 혼란: 인덱스가 너무 많으면 최적 선택 어려움

📌 실제 경험 - JASSO 프로젝트:

  • 학생 테이블에 인덱스 15개 발견 → 8개로 축소
  • 사용되지 않는 인덱스 제거 (MySQL slow query log 분석)
  • INSERT 성능 40% 향상

Q7. 대용량 데이터 조회 시 성능을 개선하는 방법은?

대용량 데이터 조회 최적화 기법:

1. 페이징 처리
  • Offset Pagination:
    • LIMIT 100 OFFSET 10000
    • 단점: Offset 클수록 느림 (앞 데이터 스캔)
  • Cursor Pagination (추천):
    • WHERE id > last_id LIMIT 100
    • 항상 일정한 성능
2. 커버링 인덱스
  • 인덱스만으로 쿼리 결과 반환 (테이블 접근 불필요)
  • SELECT id, name FROM users WHERE status='active'
  • INDEX(status, id, name) 생성 → 커버링 인덱스
3. 파티셔닝
  • Range 파티셔닝: 날짜별 (년/월)
  • Hash 파티셔닝: 균등 분산
  • 효과: 쿼리 시 필요한 파티션만 스캔
4. 쿼리 결과 캐싱
  • Redis/Memcached: 자주 조회되는 데이터
  • Spring Cache: @Cacheable 어노테이션
  • DB Query Cache: MySQL Query Cache (5.7 이하)

📌 한샘 SAP 주문 조회 시스템:

  • 문제: 전체 주문 데이터 1,000만 건, 조회 시간 10초
  • 해결:
    • 주문일자 기준 월별 파티셔닝
    • 커버링 인덱스: INDEX(주문일자, 고객ID, 주문번호, 상태)
    • Redis 캐싱: 최근 7일 주문 데이터
  • 결과: 조회 시간 10초 → 0.5초 (95% 개선)

Q9. 무중단 마이그레이션이 필요한 경우 어떻게 진행하나요?

무중단 마이그레이션 전략:

1. 이중 쓰기(Dual Write) 방식
  • Phase 1: 구 DB에만 쓰기 (현재 상태)
  • Phase 2: 구 DB + 신 DB 동시 쓰기
    • 애플리케이션 레벨에서 양쪽 모두 업데이트
    • 읽기는 여전히 구 DB에서
  • Phase 3: 과거 데이터 백그라운드 마이그레이션
  • Phase 4: 신 DB로 읽기 전환
    • 데이터 정합성 확인 후
    • Feature Toggle로 점진적 전환
  • Phase 5: 구 DB 쓰기 중단, 신 DB만 사용
2. CDC(Change Data Capture) 활용
  • 도구: Debezium, AWS DMS, Oracle GoldenGate
  • 방식:
    • 구 DB의 변경사항을 실시간 캡처
    • 신 DB에 자동 반영
    • 애플리케이션 코드 변경 최소화
3. 파란-초록 배포(Blue-Green Deployment)
  • Blue: 현재 운영 중인 구 시스템
  • Green: 신 시스템 (데이터 동기화 완료)
  • 전환: 로드밸런서 스위칭으로 즉시 전환
  • 롤백: 문제 발생 시 즉시 Blue로 복귀
4. 정합성 검증
  • 이중 쓰기 기간 중 데이터 불일치 감지 시스템
  • 주기적으로 구/신 DB 데이터 비교
  • 불일치 발견 시 알림 + 자동 동기화

📌 MSA 전환 시 적용:

  • 모놀리식 DB → 각 서비스별 독립 DB
  • 이중 쓰기 방식으로 3주간 병행 운영
  • 데이터 정합성 확인 후 신 시스템으로 완전 전환
  • 비즈니스 중단 시간: 0

Q10. 정규화와 역정규화의 차이는? 언제 역정규화를 사용하나요?

정규화(Normalization):

  • 목적: 데이터 중복 최소화, 무결성 유지
  • 1NF: 원자성 (각 컬럼은 하나의 값)
  • 2NF: 부분 함수 종속 제거
  • 3NF: 이행 함수 종속 제거
  • 장점: 데이터 일관성, 수정 이상 방지
  • 단점: JOIN 증가 → 조회 성능 저하

역정규화(Denormalization):

  • 목적: 조회 성능 향상
  • 방법: 의도적으로 중복 데이터 허용
  • 장점: JOIN 감소, 조회 속도 향상
  • 단점: 데이터 불일치 가능성, 저장 공간 증가

역정규화 사용 시나리오:

  • 읽기가 쓰기보다 압도적으로 많은 경우
    • 예: 통계 데이터, 대시보드
  • 복잡한 JOIN으로 성능 문제 발생
    • 5개 이상 테이블 JOIN → 응답시간 5초 이상
  • 집계 데이터가 자주 필요한 경우
    • 예: 주문 총액, 상품 평점 평균

📌 한샘 주문 시스템 역정규화 적용:

  • 문제: 주문 상세 페이지 로딩 5초 (ORDER, ORDER_ITEM, PRODUCT, CUSTOMER 4개 JOIN)
  • 해결: ORDER 테이블에 총 금액, 고객명, 대표 상품명 컬럼 추가 (역정규화)
  • 동기화: 주문 생성/수정 시 동기화 로직 추가
  • 결과: 조회 시간 5초 → 0.5초 (90% 개선)

Q11. 트랜잭션 격리 수준(Isolation Level)에 대해 설명해주세요.

트랜잭션 격리 수준 4단계:

1. READ UNCOMMITTED (레벨 0)
  • 커밋되지 않은 데이터도 읽기 가능
  • 문제: Dirty Read 발생 가능
  • 사용: 거의 사용 안함 (데이터 정합성 보장 불가)
2. READ COMMITTED (레벨 1) - 대부분 DBMS 기본값
  • 커밋된 데이터만 읽기
  • 해결: Dirty Read 방지
  • 문제: Non-Repeatable Read 발생 가능
    • 같은 쿼리 두 번 실행 시 다른 결과
  • 사용: Oracle, PostgreSQL 기본값
3. REPEATABLE READ (레벨 2) - MySQL InnoDB 기본값
  • 트랜잭션 시작 시점의 데이터를 일관되게 읽음
  • 해결: Non-Repeatable Read 방지
  • 문제: Phantom Read 발생 가능 (MySQL은 Next-Key Lock으로 방지)
  • 사용: 대부분의 일반적인 케이스
4. SERIALIZABLE (레벨 3)
  • 완전한 격리, 트랜잭션 순차 실행
  • 해결: Phantom Read까지 완전 방지
  • 문제: 성능 저하, 동시성 매우 낮음
  • 사용: 금융 거래 등 정합성이 극도로 중요한 경우

📌 MONEX 증권사 프로젝트:

  • 상황: 주식 거래 시스템 (동시 주문 처리)
  • 선택: REPEATABLE READ
    • READ COMMITTED: 주문 처리 중 잔고 변경 → 문제 발생
    • SERIALIZABLE: 동시성 너무 낮음 (초당 수천 건 처리 불가)
    • REPEATABLE READ: 적절한 격리 + 성능 균형
  • 추가 대책: 비관적 락(SELECT FOR UPDATE)으로 잔고 테이블 보호

📌 한샘 재고 관리:

  • 문제: 동시에 같은 상품 주문 시 재고 부족 문제
  • 해결: 재고 차감 시 비관적 락 사용
    • SELECT stock FROM product WHERE id=? FOR UPDATE
    • 재고 확인 → 차감까지 락 유지
  • 결과: 재고 부족 오류 0건

Q12. 데드락(Deadlock)이란? 어떻게 해결했나요?

데드락(Deadlock) 개념:

  • 두 개 이상의 트랜잭션이 서로의 락을 기다리며 무한 대기하는 상태
  • 예시:
    • 트랜잭션 A: 테이블 X 락 → 테이블 Y 대기
    • 트랜잭션 B: 테이블 Y 락 → 테이블 X 대기
    • → 서로 무한 대기

데드락 해결 방법:

1. 락 순서 통일
  • 모든 트랜잭션이 같은 순서로 테이블 접근
  • 예: 항상 A → B → C 순서로 락 획득
2. 트랜잭션 시간 최소화
  • 불필요한 로직을 트랜잭션 밖으로
  • 락 유지 시간 단축
3. 인덱스 활용
  • 인덱스 없으면 테이블 전체 락 발생 가능
  • WHERE 절 컬럼에 인덱스 → 행 단위 락
4. 타임아웃 설정
  • innodb_lock_wait_timeout (MySQL)
  • 일정 시간 후 자동 롤백 + 재시도
5. 낙관적 락 사용
  • Version 컬럼으로 충돌 감지
  • 충돌 발생 시에만 재시도

📌 한샘 주문 처리 시스템 데드락 해결:

  • 문제: 주문 생성 시 데드락 빈번 (주문+재고+포인트 테이블 동시 수정)
  • 원인 분석:
    • 트랜잭션 A: 주문 → 재고 → 포인트 순서
    • 트랜잭션 B: 포인트 → 재고 → 주문 순서
    • → 데드락 발생
  • 해결:
    • 모든 주문 트랜잭션의 테이블 접근 순서 통일: 주문 → 재고 → 포인트
    • 재고 조회에 인덱스 추가 (Full Table Scan 방지)
    • 트랜잭션 시간 단축: 외부 API 호출을 트랜잭션 밖으로
  • 결과: 데드락 발생 99% 감소

Q13. N+1 문제란? 어떻게 해결했나요?

N+1 문제 개념:

  • 연관된 엔티티를 조회할 때 추가 쿼리가 N번 발생하는 문제
  • 예시:
    • 주문 10개 조회 (1번 쿼리)
    • 각 주문의 상품 조회 (10번 쿼리)
    • → 총 11번 쿼리 (1 + N)

해결 방법:

1. Fetch Join (JPQL)
  • SELECT o FROM Order o JOIN FETCH o.items
  • 한번의 쿼리로 연관 데이터까지 조회
  • 주의: 페이징 시 문제 발생 가능 (메모리에서 페이징)
2. @EntityGraph (JPA)
  • @EntityGraph(attributePaths = {"items"})
  • 특정 연관관계만 Eager Loading
3. Batch Size 설정
  • @BatchSize(size = 100) (Hibernate)
  • N번 쿼리 → N/100번 쿼리 (IN 절 사용)
4. DTO Projection
  • 필요한 데이터만 조회 (엔티티 아님)
  • JOIN 쿼리 직접 작성

📌 MSA 세금계산서 프로젝트:

  • 문제: 세금계산서 100건 조회 시 201번 쿼리 (1 + 100 + 100)
    • 세금계산서 조회 (1번)
    • 각 세금계산서의 품목 조회 (100번)
    • 각 품목의 세부정보 조회 (100번)
  • 해결:
    • Fetch Join 적용: 세금계산서 + 품목 + 세부정보 한번에 조회
    • 페이징은 ID 범위로 처리
  • 결과: 201번 쿼리 → 1번 쿼리, 처리 시간 70% 단축

📌 한샘 주문 조회 시스템:

  • 문제: 주문 목록 페이지에서 1000건 조회 시 2001번 쿼리
  • 해결: @BatchSize(size = 100) 설정
    • 1000번 → 10번 쿼리 (IN 절로 100개씩 조회)
  • 결과: 페이지 로딩 시간 8초 → 1.5초

Q14. Replication과 Sharding의 차이는? 언제 사용하나요?

Replication (복제):

  • 개념: 같은 데이터를 여러 DB 서버에 복사
  • Master-Slave 구조:
    • Master: 쓰기 전담 (INSERT, UPDATE, DELETE)
    • Slave: 읽기 전담 (SELECT)
    • Master → Slave 데이터 동기화
  • 장점:
    • 읽기 성능 향상 (Slave 증설로 확장)
    • 고가용성 (Master 장애 시 Slave 승격)
    • 백업 용이
  • 단점:
    • 쓰기는 여전히 Master 한대 (쓰기 확장 불가)
    • Replication Lag (마스터-슬레이브 간 지연)
  • 사용 시나리오: 읽기가 쓰기보다 훨씬 많은 경우 (10:1 이상)

Sharding (샤딩):

  • 개념: 데이터를 여러 DB 서버에 분산 저장
  • 분산 방식:
    • Range Sharding: ID 1-1000 → Shard1, 1001-2000 → Shard2
    • Hash Sharding: user_id % 4 → Shard 0~3
    • Geography Sharding: 지역별 (한국, 일본, 미국)
  • 장점:
    • 쓰기/읽기 모두 확장 가능
    • 대용량 데이터 처리 (수억 건 이상)
  • 단점:
    • 구현 복잡도 높음
    • Shard 간 JOIN 불가능
    • Shard 재분배 어려움
  • 사용 시나리오: 데이터가 수억 건 이상, 쓰기 트래픽도 많은 경우

📌 실제 경험:

  • 한샘 주문 시스템: Master-Slave Replication 구성
    • 주문 조회(읽기 90%) → Slave 3대
    • 주문 생성(쓰기 10%) → Master 1대
    • 읽기 성능 3배 향상
  • Credit Saison (일본 프로젝트): 데이터 마이그레이션 시 샤딩 고려
    • 2,500만 건 카드 거래 데이터
    • 월별 파티셔닝으로 유사 샤딩 효과

Q15. 실행 계획(Explain Plan)을 어떻게 분석하고 개선했나요?

실행 계획 분석 핵심 항목:

1. Type (MySQL) / Access Method
  • system: 테이블에 레코드가 1개 (최상)
  • const: PK/UNIQUE로 조회 (매우 좋음)
  • eq_ref: JOIN 시 PK/UNIQUE 사용 (좋음)
  • ref: 일반 인덱스 사용 (양호)
  • range: 인덱스 범위 스캔 (보통)
  • index: 인덱스 풀 스캔 (나쁨)
  • ALL: 테이블 풀 스캔 (최악) → 반드시 개선 필요
2. Rows / Cardinality
  • 예상 스캔 행 수
  • 높을수록 성능 저하
  • 인덱스 추가로 rows 감소
3. Extra
  • Using index: 커버링 인덱스 (좋음)
  • Using where: WHERE 필터링 (보통)
  • Using filesort: 정렬 시 임시 파일 사용 (나쁨) → 인덱스로 정렬 해결
  • Using temporary: 임시 테이블 생성 (나쁨) → GROUP BY 최적화

📌 실제 개선 사례 - 한샘 주문 조회:

  • 문제 쿼리: 
    SELECT * FROM orders
WHERE customer_id = 12345
  AND order_date BETWEEN '2024-01-01' AND '2024-12-31'
ORDER BY order_date DESC;
  • EXPLAIN 결과 (개선 전):
    • type: ALL (테이블 풀 스캔)
    • rows: 1,000,000 (백만 건 스캔)
    • Extra: Using filesort (정렬에 임시 파일 사용)
    • 실행 시간: 8초
  • 개선 방법:
    • 복합 인덱스 생성: INDEX(customer_id, order_date DESC)
    • customer_id 먼저 (카디널리티 높음)
    • order_date DESC: 정렬 방향까지 인덱스에 포함
  • EXPLAIN 결과 (개선 후):
    • type: range (인덱스 범위 스캔)
    • rows: 1,200 (필터링된 결과만)
